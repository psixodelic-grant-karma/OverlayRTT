# **Техническое задание: Система экранного перевода в реальном времени (Real-Time Screen Translator)**

## **1\. Общее описание**

Разработать кроссплатформенное (Windows/Linux) приложение для захвата области экрана, распознавания текста (OCR), его перевода с использованием локальных LLM и отображения переведённого текста в виде настраиваемого оверлея. Ключевое требование — **минимальное влияние на производительность** целевого приложения (игры, редактора, видео, браузер) и **асинхронная работа** всех модулей.

Создание высокопроизводительного приложения на C++ для перевода текста с экрана в реальном времени с использованием локальных моделей (OCR + LLM) и отображением результата поверх исходного контента или в отдельном окне.

## **2\. Функциональные блоки (4 модуля)**

Система состоит из 4 независимых асинхронных модулей:

1.  Захват экрана (Capture)
2.  Распознавание текста (OCR)
3.  Перевод (Translation)
4.  Отрисовка (Overlay Renderer)

- дополнительный компонент:

1.  Лаунчер / UI (Python)

### **2.1. Модуль захвата экрана (Capture)**

**Назначение:** получить сырое изображение из GPU-памяти с минимальной задержкой.

**Возможности:**

- Захват всей области экрана.
- Захват конкретного окна / приложения (по HWND или PID).
- Захват пользовательской области (прямоугольник, в т.ч. несколько областей).
- **Игнорирование областей** (задаваемые прямоугольники не захватываются → экономия ресурсов).
- Копирование кадра напрямую из памяти GPU (без лишнего копирования в RAM).
- Режимы запуска захвата:
    - Callback-функция по событию (например, изменение содержимого окна).
    - Таймер с заданным интервалом.
    - **По умолчанию:** callback + таймаут (при отсутствии изменений — принудительный опрос).
- Поддержка бэкендов (выбираются в лаунчере):
    - DXGI (Windows)
    - Vulkan, OpenGL (кроссплатформа)
    - screen_capture_lite
    - FFmpeg (как fallback)

**Выходные данные:** кадр в формате RGBA/NV12 с меткой времени и координатами на экране.

struct Frame {  
ImageBuffer buffer;  
Timestamp ts;  
Region\[\] regions;  
};

### **2.2. Модуль распознавания текста (OCR)**

**Назначение:** найти все текстовые блоки на кадре, извлечь текст, координаты, параметры шрифта/фона.

**Требования:**

- **Сегментация текста на контекстные блоки** – текст не сливается в один абзац, если он визуально разделён (разные кнопки, диалоги, заголовки).
- Для каждого блока определяется:
    - Текст (строка или несколько строк, если они связаны).
    - Прямоугольник ограничивающей рамки (x, y, w, h).
    - Фон под текстом (преобладающий цвет или текстура).
    - Параметры исходного шрифта (размер, гарнитура, цвет, начертание – насколько позволяет OCR).
- Используемые движки: Tesseract, OCRopus, PaddleOCR (локально, без облачных вызовов).
- **Предобработка изображения** перед OCR (настраивается):
    - Масштабирование, бинаризация, устранение шумов, повышение резкости, изменение контраста.
    - сжатие без потери важной информации.
- **Производительность:** блок должен успевать обрабатывать кадры асинхронно, не дожидаясь перевода и отрисовки.
- Трекинг текста если текст не меняется → не перерабатывать OCR

**Выходные данные:** список TextBlock (текст, координаты, метаданные шрифта/фона, уникальный ID блока).

struct TextBlock {  
std::string text;  
Rect bbox;  
Color background;  
FontInfo font;

.....

};

### **2.3. Модуль перевода (Translator)**

**Назначение:** локальный перевод каждого контекстного блока.

**Ключевые требования:**

- **Локальная работа** – без отправки данных в сеть.
- Использование LLM (например, LLaMA, GPT4All, Ollama, CTranslate2) или библиотек типа LibreTranslate (локально) Marian NMT llama.cpp (локальные LLM) Argos Translate.
- **Автоопределение исходного языка** (или ручное указание).
- **Язык перевода** задаётся пользователем (например, ru, en, zh).
- **Раздельный перевод блоков** – если блоки не связаны (разные UI-элементы), они переводятся независимо. Связанный текст (один блок, многострочный) переводится как единое целое.
- **Многопоточность** – разные блоки могут переводиться параллельно (пул потоков).
- **Кэширование перевода** одинаковые строки не переводятся повторно
- Вход: TextBlock.text. Выход: переведённая строка (привязка к тому же block_id).

**Дополнительно:**

- Мультиязычный текст внутри одного блока – либо определение основного языка, либо пропуск.

### **2.4. Модуль оверлея (Overlay)**

**Назначение:** отрисовать переведённый текст поверх исходного приложения, не мешая взаимодействию.

**Требования к оверлею:**

- **Не перехватывает ввод** мыши/клавиатуры (кроме глобальных горячих клавиш).
- Работает через OpenGL, Vulkan или WinAPI прозрачное окно (WS_EX_LAYERED, WS_EX_TRANSPARENT).
- Отрисовывается только то, что нужно — поверх областей, где был распознан исходный текст.
- Для каждого переведённого блока создаётся **подложка** (фон под текстом), чтобы перевод был читаемым.

**Настройки подложки:**

|     |     |
| --- | --- |
| **Режим** | **Описание** |
| Фиксированный цвет + прозрачность | RGB + Alpha (0..255) |
| Авто-фон | Вычисляется средний цвет пикселей в области исходного текста (из кадра) и используется как фон подложки |
| Градиентный авто-фон | Градиент от цвета левого края к правому (Вычисляется средний цвет в областях и делаем переливание делаем частично мутным матовым прозрачным) |
| Оригинальный фон ("умный ластик") | Сначала "стирается" исходный текст (inpainting) – восстанавливается фон, затем поверх пишется перевод |

**Настройки самого текста:**

- Цвет, шрифт, размер, начертание – задаются вручную.
- **Авторежим шрифта:** извлечь параметры исходного шрифта (высота, ширина, цвет, гарнитура) из блока OCR и применить к переводу (с возможным масштабированием, т.к. длина перевода отличается).

**Дополнительная опция:** вместо оверлея выводить перевод в отдельное окно (настраивается).

## **3\. Архитектура и асинхронный конвейер (Throttle + Pending Frame)**

### **3.1. Основные принципы**

Система строится как конвейер из четырёх независимых модулей: **Capture → OCR → Translator → Overlay**.  
В отличие от классических конвейеров с вытеснением (отменой задач), здесь используется **механизм «один активный кадр + один ожидающий» (single active + single pending)**.

**Правила для каждого модуля-обработчика (OCR, Translator):**

1.  В каждый момент времени модуль может выполнять **не более одной задачи** (обработка кадра или блока).
2.  Если во время выполнения задачи поступают новые входные данные (новый кадр для OCR, новый список блоков для Translator), они **не отменяют текущую задачу**, а сохраняются как **pending-данные** (только самые свежие; предыдущий pending перезаписывается).
3.  После завершения текущей задачи модуль проверяет наличие pending-данных:
    - Если есть → немедленно запускает новую задачу с этими данными.
    - Если нет → переходит в спящий режим до появления следующих данных.
4.  **Никаких отмен, прерываний или вытеснений.** Любая начатая задача обязательно доводится до конца, даже если её входные данные устарели.

### **3.2. Структура очередей**

Очереди между модулями — **потокобезопасные контейнеры с одним слотом** (single-slot queue) или с перезаписываемым последним элементом. Для каждого канала данных используется такой тип:

text

template&lt;typename T&gt;class LatestQueue { std::atomic&lt;bool&gt; has_pending{false}; T pending_value; std::mutex mtx;public: void push(const T& value) { std::lock_guard lock(mtx); pending_value = value; has_pending.store(true, std::memory_order_release); } bool pop(T& out) { if (!has_pending.load(std::memory_order_acquire)) return false; std::lock_guard lock(mtx); if (!has_pending.load()) return false; out = std::move(pending_value); has_pending.store(false); return true; }};

**Важно:**

- Между Capture и OCR передаётся **только один актуальный кадр** (размером до 8 МБ для 1080p RGBA).
- Между OCR и Translator передаётся **только один список TextBlock** (самый свежий результат распознавания).
- Между Translator и Overlay — **только один список TranslatedBlock**.

### **3.3. Асинхронный цикл модуля (псевдокод)**

cpp

void ocr_thread() { Frame current_frame; bool has_pending = false; Frame pending_frame; while (running) { // 1. Если нет активной задачи, пытаемся взять новый кадр if (!has_active_task) { if (capture_queue.pop(current_frame)) { has_active_task = true; // 2. Запускаем OCR синхронно (без отмен) auto result = run_ocr(current_frame); // 3. Отправляем результат в Translator translator_queue.push(result); has_active_task = false; // 4. После завершения проверяем, не появился ли pending if (capture_queue.pop(pending_frame)) { current_frame = pending_frame; has_active_task = true; // повторяем цикл (немедленно) continue; } } } // 5. Если активная задача есть или нет новых данных – спим wait_for_event_or_timeout(); }}

Аналогично для модуля Translator.

### **3.4. Тайминги и гарантии задержки**

**Допущения:**

- Capture выдает кадры с интервалом T_capture (например, 16.6 мс при 60 FPS).
- OCR обрабатывает кадр за T_ocr (например, 50 мс).
- Translator обрабатывает блок за T_trans (например, 200 мс).

#### **3.4.1. Максимальная задержка между появлением текста и отрисовкой перевода**

В худшем случае, когда модуль только начал обработку устаревших данных и сразу после этого приходит свежий кадр:

- Текст появился в кадре C_new в момент t0.
- Модуль OCR занят обработкой кадра C_old до момента t0 + T_ocr.
- После завершения OCR видит pending = C_new и запускает его обработку.  
    Результат для C_new будет готов в t0 + T_ocr + T_ocr (две полных обработки OCR).
- Затем Translator аналогично может добавить 2 \* T_trans, если тоже занят.

**Верхняя граница задержки (P99):  
**Delay_max = 2 \* (T_ocr + T_trans) + T_capture  
Для примера: 2\*(50+200) + 16 ≈ 516 мс.

#### **3.4.2. Типичная задержка (установившийся режим)**

При стабильном потоке кадров (60 FPS) и T_ocr = 50 мс, T_trans = 200 мс:

- OCR обрабатывает кадры C0, C2, C4, ... (каждый второй, так как T_ocr > T_capture).  
    Каждый результат соответствует кадру, который был актуален примерно 0.5 \* T_ocr + T_capture назад.  
    Средняя задержка OCR ≈ 30–40 мс плюс время передачи.
- Translator работает аналогично, но его T_trans больше, поэтому он будет обрабатывать блоки от каждого второго кадра OCR.  
    Итоговая **типичная задержка** (P50) ≈ T_ocr + T_trans + T_capture = 50+200+16 ≈ **266 мс**.

Это практически совпадает с задержкой в исходном ТЗ с вытеснением, но **без риска голодания** (когда OCR никогда не завершается).

#### **3.4.3. Гарантия отсутствия бесконечных отмен**

В исходном ТЗ при частоте кадров выше пропускной способности OCR (например, 120 FPS при T_ocr=50 мс) модуль мог постоянно перезапускаться и никогда не выдать результат.  
В новом механизме **каждая начатая задача обязательно завершается**, поэтому система всегда выдаёт перевод для каждого второго или третьего кадра, даже при сверхвысокой частоте захвата.

### **3.5. Поведение при изменении нагрузки**

- **Если текст не меняется** (стабильные кадры): OCR будет обрабатывать одни и те же кадры многократно. Чтобы избежать этого, модуль OCR обязан реализовать **трекинг изменений** (сравнение хэша или карты различий). При отсутствии изменений OCR не запускается, а просто повторно отправляет предыдущий результат.
- **Если Translator сильно медленнее OCR** (например, 500 мс против 50 мс):  
    Очередь между OCR и Translator будет накапливать только один pending-список блоков. Это означает, что Translator всегда будет обрабатывать самый свежий результат OCR, игнорируя промежуточные. Задержка перевода при этом не превышает T_trans + T_ocr.

### **3.6. Преимущества перед вытеснением (отменой задач)**

|     |     |     |
| --- | --- | --- |
| **Критерий** | **Вытеснение (исходное ТЗ)** | **Throttle + pending frame (новое)** |
| Потеря работы | Частичная (отменённые такты) | Нулевая (все вычисления завершены) |
| Риск голодания (live lock) | Высокий при частоте > 1/T_ocr | Отсутствует |
| Задержка в худшем случае | Неограниченная (может не быть результата) | ≤ 2\*(T_ocr+T_trans) |
| Простота реализации | Сложно (кооперативная отмена, очистка ресурсов) | Просто (нет отмен, только очереди) |
| Поведение в играх (быстрый скролл) | Мигание, пропуски | Стабильный перевод с небольшим отставанием |

### **3.7. Особенности для модуля Translator**

Модуль перевода работает с отдельными текстовыми блоками, а не с целыми кадрами. Принцип «один активный + один pending» применяется **к каждому блоку индивидуально** (или к пулу блоков одного кадра). Рекомендуемая реализация:

- Translator получает от OCR список блоков \[B1, B2, ..., Bn\].
- Для каждого блока запускается задача перевода в пуле потоков. Количество одновременно выполняемых задач ограничено (например, числом ядер CPU).
- Если для одного и того же block_id (уникальный идентификатор, вычисленный OCR по позиции и тексту) поступает новый текст, а старый ещё переводится, то старый перевод **не отменяется**, а его результат просто игнорируется (при завершении проверяется, не появился ли более свежий pending для этого блока). Сам же поток переводчика не прерывается.

Это даёт баланс: долгий перевод не мешает начать перевод более свежего текста, но при этом не теряется уже проделанная работа (она может пригодиться, если новый текст окажется таким же).

### **3.8. Интеграция с лаунчером (конфигурация)**

В GUI лаунчера должны быть настройки, связанные с этим механизмом:

- **Режим конвейера** (выбор между «Throttle + pending» и классическим вытеснением) — по умолчанию throttle.
- **Максимальное количество pending-кадров** (1 – как описано выше, но можно разрешить 2–3 для очень медленных OCR).
- **Таймаут ожидания** для спящего потока (по умолчанию 10 мс, чтобы быстро реагировать на новые данные).

### **3.9. Исключения и сбои**

- Если OCR не может разобрать текст (ошибка), то на выход отправляется пустой список блоков. Pending-кадр при этом не теряется.
- Если Translator падает (исключение), модуль перезапускается, а последний успешный перевод остаётся в очереди для Overlay. Pending-данные сохраняются.
- При долгом зависании модуля (более timeout = 5 секунд) система логирует ошибку и принудительно сбрасывает активную задачу, переходя к pending. Это аварийный случай, но он предусмотрен.

### **3.10. Резюме**

Предлагаемый конвейер на основе **throttle + pending frame** является более предсказуемым, устойчивым к перегрузкам и простым в реализации по сравнению с вытеснением. Он гарантирует, что пользователь всегда увидит перевод (пусть и с небольшой задержкой), а система не будет бесконечно перезапускать задачи впустую. Это оптимальный выбор для приложения реального времени с требованием «минимальное влияние на производительность» и «стабильная работа при любом FPS захвата».

_Примечание:_ в исходном ТЗ (раздел 2) функциональные блоки не меняются. Изменяется только механизм их взаимодействия, описанный выше. Все настройки модулей, перечисленные в разделе 4, остаются в силе.

## **4\. Лаунчер и конфигурация (GUI на Python / C++)**

Лаунчер — отдельное приложение (можно на Python с PyQt/PySide + современный дизайн), которое позволяет:

### **4.1. Настройка блока Capture**

- Выбор бэкенда: OpenGL, Vulkan, DXGI, screen_capture_lite, FFmpeg.
- Выбор источника: весь экран, конкретное окно, область (координаты), области игнорирования.
- Качество скриншота (разрешение, формат пикселей, сжатие).
- Интервал таймера (мс) или включение callback-режима.

### **4.2. Настройка блока OCR**

- Движок: Tesseract, PaddleOCR (локально) – выбор.
- Язык распознавания (например, eng+rus+deu).
- Фильтры предобработки (гаусс, медианный, повышение резкости, бинаризация Otsu).
- Алгоритм сжатия (если требуется) – PNG, JPEG (качество 90%+).
- Минимальный размер текста (отсев мелкого шума).

### **4.3. Настройка блока Translator**

- Модель LLM (путь к файлу .gguf или bin).
- Язык исходный (auto / задать явно) и язык перевода.
- Количество потоков перевода.
- Настройки инференса (temp, top-p, контекстное окно).

### **4.4. Настройка блока Overlay**

- Режим фона (фикс. цвет + alpha / авто / градиент / "умный ластик").
- Режим текста (цвет, шрифт / авто-стиль).
- Позиционирование: точно поверх исходного текста или смещённое (на случай, если перевод длиннее).
- Горячие клавиши: вкл/выкл оверлей, временная пауза захвата, принудительный захват кадра.

### **4.5. Интерфейс управления пакетами (Plugin Manager)**

Система **модульная** – ядро (диспетчер очередей, интерфейсы модулей) общее. Каждый из четырёх модулей подключается как плагин через единый интерфейс (C++ абстрактный класс или FFI).

- Plug & Play
- Если модуль отсутствует:
- приложение НЕ падает
- предлагает установить

**Пакетный менеджер в лаунчере:**

- Отображает список установленных модулей и доступных пакетов (бэкенды захвата, языковые данные для OCR, языковые модели для переводчика, движки оверлея).
- При попытке активировать модуль, если нужные ресурсы отсутствуют:
    - Программа **не падает**, а показывает сообщение: _"Модуль X требует пакет Y. Установить?"_.
    - Автоматически скачивает и устанавливает пакет (из заданного репозитория) через встроенный установщик.
- Пример минимальной конфигурации для работы:
    - Capture: DXGI (Win) или screen_capture_lite (Linux).
    - OCR: Tesseract + языковой пакет eng.
    - Translator: локальная LLM en-ru (например, модель на 500МБ).
    - Overlay: OpenGL (минимальный).

## **5\. Технические требования к реализации**

### **5.1. Производительность и язык**

- **Ядро и модули:** C++17/20, максимальная эффективность, прямой доступ к GPU-памяти, SIMD для предобработки.
- **Лаунчер:** Python + Qt (для быстрого прототипирования и управления пакетами) либо тоже C++ с Qt.
- **Асинхронность:** std::thread, boost::lockfree::queue или moodycamel::ConcurrentQueue.

### **5.2. Требования к памяти и CPU**

- Захват кадра + OCR не должны загружать более одного ядра CPU на 100% постоянно.
- При паузе захвата или отсутствии изменений на экране – потребление ресурсов близко к нулю (опрос с низкой частотой или ожидание события).

### **5.3. Совместимость**

- Windows 10/11, Linux (X11/Wayland – с поддержкой screencapture через PipeWire).
- Игры на полном экране (захват через DXGI или Vulkan overlay).
- Приложения с аппаратным ускорением (браузеры, видеоредакторы).

## **6\. Исключения и обработка ошибок**

- Если OCR не может разобрать текст – блок не отправляется на перевод.
- Если LLM не отвечает (высокая задержка) – используется последний удачный перевод или ничего не рисуется.
- При сбое одного модуля (например, захват потерял окно) – система переходит в режим ожидания и пытается восстановить соединение без перезапуска всей программы.
